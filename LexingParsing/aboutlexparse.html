Basics

Lexing and Parsing is the way to make computer programming languages. They actually tell the computer what to do which the computer then takes those instructions and executes them. When finished with the Lexing/Parsing, the computer then sends it to a interpreter or compiler, which then executes the code. Interpreted is usually slower than compiled, but it depends.

Interpreted

Here are some pros and cons of Interpreted langauges: Pros:

  • Interpreted languages tend to be more high-level, meaning, easier for the humans to read. While low-level means it is easier for the computers to read.
  • Interpreted languages can catch errors faster, and tell you about them, instead of having to recompile every time.
  • Interpreted Languages are usually better for beginners, even though some languages are better to learn before others.

    • Cons:
  • They are slow.

    • Compiled langauges are here:

    Pros:

  • Fast, super fast
  • Used for great languages

    • Cons:

  • Tend to be more low-level
  • Cannot find errors as well as interpreters.


    • Back to lexing and parsing

    So we will start off with the topic lexing, because it comes first in the programming of a language. A Lexer, takes source code, and then lexes them down into a bunch of tokens . An example would be:
    Source code:

    print("Hello, world!")

    After going through the pre -lexer it would be this:

    INITIALIZER "print"
    OPEN_PAREN '('
    DOUBLEQOUTE '"'
    STRING "Hello, world!"
    DOUBLEQOUTE '"'
    CLOSED_PAREN ')'

    Then that would go to the lexer, saying:

    INITIALZER "print"
    OPEN_PAREN '('
    STRING "Hello, world!"
    CLOSED_PAREN ')'

    But you could also make an all in one lexer, which does the quote stuff like this(easier way):
    Everything after the " is equal to a string, and when a second " is called that is the end of the string.
    The basic process:


    src -> tokenizing/lexing -> parsing -> compiling/interpreting -> output(your screen/terminal)
